# ls-catch: 프로젝트 현황 파악 및 컨텍스트 복구

새로운 대화를 시작했을 때, 이전까지의 작업 현황을 빠르게 파악하여 바로 작업에 들어갈 수 있도록 합니다.

## 실행 절차

### 1단계: 정보 수집

다음 정보를 **병렬로** 수집합니다:

1. **git log** - 최근 커밋 10~20개를 확인합니다. 커밋 메시지의 "완료된 작업", "다음 진행 사항" 섹션이 있으면 중점적으로 파악합니다.
2. **git status** - 현재 브랜치, 커밋되지 않은 변경사항 확인 (`-uall` 플래그 사용 금지)
3. **git diff** - 작업 중이던 미커밋 변경사항 확인
4. **계획서** - `.claude/plans/` 디렉토리의 모든 계획서를 읽어서 Phase별 진행 상태 파악
5. **메모리** - `/Users/jyj/.claude/projects/-Users-jyj-ls-projects-lecture-0407/memory/MEMORY.md` 확인

### 2단계: 현황 분석 및 보고

수집한 정보를 아래 형식으로 **한국어**로 정리하여 사용자에게 보고합니다:

```
## 프로젝트 현황 리포트

### 브랜치 상태
- 현재 브랜치: {브랜치명}
- 미커밋 변경: {있음/없음} (있으면 변경 파일 목록)

### 진행 중인 계획서
- 계획서: {파일명}
- 상태: {Phase별 진행 현황 요약}

### 최근 완료된 작업
가장 최근 커밋부터 역순으로 주요 작업 내역을 정리합니다.
- {날짜} {커밋 요약}: {핵심 내용}
- {날짜} {커밋 요약}: {핵심 내용}
  ...

### 미완료 작업 (다음에 해야 할 것)
가장 최근 커밋의 "다음 진행 사항"과 계획서의 미완료 항목을 합쳐서 정리합니다.
우선순위가 높은 것부터 나열합니다.
- [ ] {작업 1} (출처: 커밋/계획서)
- [ ] {작업 2} (출처: 커밋/계획서)
  ...

### 주의사항
이전 작업에서 언급된 알려진 이슈, 제약사항, 주의할 점을 정리합니다.
- {주의사항 1}
- {주의사항 2}
  ...

### 현재 작업 중인 것
미커밋 변경사항이 있다면, 어떤 작업을 하다가 멈춘 것인지 분석합니다.
없으면 "없음 - 새 작업 시작 가능" 으로 표시합니다.
```

### 3단계: 다음 행동 제안

현황 보고 후, 사용자에게 다음 행동을 제안합니다:
- 미커밋 변경이 있으면: "커밋 먼저 하시겠어요, 아니면 이어서 작업할까요?"
- 미완료 작업이 있으면: "가장 우선순위 높은 작업인 {작업명}부터 시작할까요?"
- 계획서가 미완료면: "계획서 {Phase}부터 이어서 진행할까요?"

---

## 주의사항

1. **커밋 메시지 파싱**: 커밋 메시지에 "## 다음 진행 사항" 또는 "## 주의사항" 섹션이 있으면 반드시 추출합니다.
2. **계획서 우선**: 계획서의 Phase 상태가 가장 신뢰할 수 있는 진행 현황입니다. 커밋과 충돌 시 계획서 기준으로 판단합니다.
3. **간결하게**: 정보가 많더라도 핵심만 추려서 보고합니다. 사용자가 30초 안에 현재 상황을 파악할 수 있어야 합니다.
4. **코드 읽지 않기**: 이 단계에서는 소스 코드를 읽지 않습니다. git 히스토리와 계획서만으로 판단합니다.
5. **한국어 사용**: 모든 출력은 한국어로 작성합니다.
